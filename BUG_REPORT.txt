BUG #1: PNG IMAGE CONVERSION AT LOW QUALITY CAUSES POTENTIAL ISSUES
================================================================================
SEVERITY: MEDIUM
LOCATION: app.py, compress_image() function, lines 161-167

CODE:
    if quality < 70 and img.mode in ("RGB", "RGBA"):
        img = img.convert("RGBA").convert(
            "P", palette=Image.ADAPTIVE, colors=256
        )

ISSUE:
    When quality < 70, the code converts the image to palette mode ("P") with 
    256 colors. However:
    
    1. Converting from RGBA to palette mode may lose or incorrectly handle the 
       alpha channel.
    2. This conversion is only applied to RGB/RGBA but not to other image modes.
    3. The conversion happens AFTER format detection, so the saved format claims
       to be PNG, but palette mode has different properties than expected.
    4. No error handling for potential issues with the palette conversion.

IMPACT:
    - Images with transparency (RGBA) may lose quality unexpectedly
    - Resulting PNG files may not display as intended
    - Users may get unexpected results without warning

EXAMPLE:
    A PNG with transparency saved at quality=60 will be converted to 256-color
    palette mode, potentially losing color depth and transparency information.

TEST RESULT: Passes without error but produces unexpected results

RECOMMENDED FIX:
    1. Add error handling for palette conversion
    2. Log a warning when palette conversion is applied
    3. Consider preserving alpha channel:
       img = img.convert("RGBA").quantize(colors=256)
       instead of converting to "P" mode
    4. Add validation that the resulting image maintains expected properties


================================================================================
BUG #2: BMP FILES DO NOT COMPRESS (0% COMPRESSION)
================================================================================
SEVERITY: LOW (Expected behavior but confusing for users)
LOCATION: app.py, compress_image() function, lines 172-175

CODE:
    else:  # BMP and anything else -> fall back to optimized JPEG-like save
        save_kwargs = {"optimize": True}

ISSUE:
    BMP is an uncompressed format. The code uses "optimize": True, but PIL
    ignores this parameter for BMP format. Result: 0% compression.
    
    - Original: 1,440,054 bytes
    - Compressed: 1,440,054 bytes (0% saved)
    
    Users may be confused when they upload BMP files and see 0% compression.

IMPACT:
    - Misleading compression statistics for BMP files
    - Wasted storage space - BMP files are never compressed
    - Poor user experience for BMP file uploads

TEST RESULT: Confirmed - BMP files show 0% compression

RECOMMENDED FIX:
    1. Convert BMP to JPEG automatically during compression
    2. Add warning message for unsupported formats (BMP, etc.)
    3. Document that BMP will not compress
    4. Example fix:
       else:  # BMP and anything else -> convert to JPEG
           fmt = "JPEG"
           ext = "jpg"
           # ... rest of JPEG compression code


================================================================================
BUG #3: MISSING QUALITY VALIDATION IN FORM INPUT
================================================================================
SEVERITY: LOW
LOCATION: app.py, upload() function, lines 281-285

CODE:
    quality = request.form.get("quality", 75)
    try:
        quality = max(1, min(95, int(quality)))
    except ValueError:
        quality = 75

ISSUE:
    While the code does validate the quality value, there's no guarantee that
    the HTML form won't send invalid values. If the form is bypassed (via
    direct API calls), non-integer values could be sent.
    
    Current handling catches ValueError but silently defaults to 75.
    No user feedback that invalid input was provided.

IMPACT:
    - Silent failure when invalid quality is provided
    - Users won't know their custom quality setting was rejected
    - No logging of invalid inputs for debugging

SEVERITY ASSESSMENT: LOW - The code handles it gracefully, but user feedback
could be improved.

RECOMMENDED FIX:
    quality = request.form.get("quality", 75)
    try:
        quality = int(quality)
        if not (1 <= quality <= 95):
            flash("Quality must be between 1 and 95. Using default (75).", "warning")
            quality = 75
    except ValueError:
        flash("Invalid quality value. Using default (75).", "warning")
        quality = 75


================================================================================
BUG #4: NO ERROR HANDLING FOR CORRUPTED IMAGE FILES
================================================================================
SEVERITY: MEDIUM
LOCATION: app.py, compress_image() function, line 145

CODE:
    img = Image.open(file_storage)

ISSUE:
    The Image.open() call could fail if:
    1. File is corrupted
    2. File is not a valid image (wrong extension)
    3. File is truncated
    
    Current error handling in upload():
        except Exception as exc:
            flash(f"Could not process that image: {exc}", "error")
    
    This catches all exceptions, which is good, but the error message could
    be more specific. Users get cryptic PIL error messages.

TEST RESULT: Generic error handling works but error messages are unclear

RECOMMENDED FIX:
    try:
        img = Image.open(file_storage)
        img.load()  # Force load to catch corruption
    except Image.UnidentifiedImageError:
        raise ValueError("File is not a valid image")
    except Image.DecompressionBombError:
        raise ValueError("Image is too large or malformed")


================================================================================
BUG #5: NO SECURITY VALIDATION ON UPLOAD SIZE
================================================================================
SEVERITY: MEDIUM
LOCATION: app.py, config section, line 39

CODE:
    MAX_CONTENT_LENGTH = 20 * 1024 * 1024  # 20 MB per upload

ISSUE:
    While MAX_CONTENT_LENGTH is set, there's no:
    1. Per-user upload limits
    2. Storage quota enforcement
    3. Protection against storage exhaustion
    4. Cleanup of old files
    5. Rate limiting on uploads
    
    A malicious user could:
    - Upload 20MB file repeatedly, exhausting storage
    - Fill the server disk with compressed images
    
IMPACT:
    - Potential Denial of Service (DOS) via disk space exhaustion
    - No cleanup mechanism for old uploads
    - No storage quota per user

RECOMMENDED FIX:
    1. Add per-user storage quota (e.g., 500MB)
    2. Add automatic cleanup of old images (>30 days)
    3. Implement rate limiting (max 5 uploads per minute)
    4. Monitor disk space and alert when low
    5. Add user quota display in dashboard


================================================================================
BUG #6: DATABASE SESSION NOT CLOSED ON ERRORS
================================================================================
SEVERITY: LOW
LOCATION: app.py, various routes

ISSUE:
    While Flask-SQLAlchemy handles session cleanup automatically, the code
    doesn't explicitly handle rollback on errors in some cases.
    
    Example in download() and delete():
        if not record or record.user_id != current_user.id:
            abort(404)

    If abort() is called, the database session might not be properly cleaned up.

IMPACT:
    - Potential database connection leaks
    - Open transactions might persist

RECOMMENDED FIX:
    Use Flask-SQLAlchemy's context manager or ensure proper cleanup:
    
    @app.teardown_appcontext
    def shutdown_session(exception=None):
        db.session.remove()

================================================================================
BUG #8: MISSING CSRF PROTECTION
================================================================================
SEVERITY: HIGH
LOCATION: app.py - ALL POST ROUTES

ISSUE:
    The Flask app doesn't use Flask-WTF for CSRF protection.
    All POST routes are vulnerable to Cross-Site Request Forgery attacks:
    - /register
    - /login
    - /upload
    - /delete
    
    An attacker could craft a malicious website that tricks users into:
    - Creating accounts
    - Uploading images
    - Deleting images

IMPACT:
    - Critical security vulnerability
    - Unauthorized actions on behalf of logged-in users
    - Data loss or unauthorized image uploads

RECOMMENDED FIX:
    1. Install Flask-WTF: pip install Flask-WTF
    2. Add to app.py:
       from flask_wtf.csrf import CSRFProtect
       csrf = CSRFProtect(app)
    3. Add CSRF tokens to all forms in templates


================================================================================
BUG #9: NO RATE LIMITING ON LOGIN (BRUTE FORCE VULNERABILITY)
================================================================================
SEVERITY: HIGH
LOCATION: app.py, login() function, lines 217-235

ISSUE:
    There's no rate limiting on login attempts. An attacker can:
    1. Perform unlimited brute force attacks
    2. Try all possible passwords without delay
    3. Perform distributed denial of service on login

IMPACT:
    - Password compromise via brute force
    - Account takeover
    - Denial of service

RECOMMENDED FIX:
    Install Flask-Limiter:
    pip install Flask-Limiter
    
    Then add:
    from flask_limiter import Limiter
    limiter = Limiter(app, key_func=lambda: request.remote_addr)
    
    @app.route("/login", methods=["GET", "POST"])
    @limiter.limit("5 per minute")
    def login():
        # ... existing code

BUG #10: NO SESSION TIMEOUT
================================================================================
SEVERITY: MEDIUM
LOCATION: app.py - config section

ISSUE:
    Flask-Login sessions don't have a timeout. A user logged in on a public
    computer could have their session hijacked indefinitely.

IMPACT:
    - Session hijacking vulnerability
    - Long-lived sessions are security risk
    - No automatic logout

RECOMMENDED FIX:
    Add to app.py config:
    app.config['PERMANENT_SESSION_LIFETIME'] = timedelta(minutes=30)
    app.config['SESSION_REFRESH_EACH_REQUEST'] = True
